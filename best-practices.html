Codex operating habits

Best practices for working with Codex

Use these habits to keep long projects clear, reviewable, and governed from first idea to finished work.

Source Note

This page is organized around public NoDrift usage guidance. Jason Liu's paper Codex-maxxing, dated 2026/05/10, is cited where it supports or extends a practice.

User habits

What the user does

The user separates planning from execution, uses voice when helpful, approves one task at a time, and keeps projects in clean workstreams. The user also questions anything that does not sound right and decides whether affected work should stop for direction.

Codex behavior

What NoDrift trains Codex to do

Codex should ask before acting, preserve approved wording, verify before reporting completion, and stop before external or destructive actions. NoDrift adds buyer-only continuity and correction discipline while keeping its internal operating language private. NoDrift does not add its files to outgoing user messages; it trains the work session to handle received output and records correctly.

Beginner

Start With Clear NoDrift Control Habits

These practices make Codex easier to guide before the work becomes complex.

User habit

Use voice for planning and review

Speak naturally when explaining a project, reviewing a page, or answering setup questions. Voice captures more context than short typed prompts.

Primary source: NoDrift private buyer materials. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

User habit

Read the AI's writing actively

Do not only wait for the answer. Read how Codex explains the work, structures sentences, names decisions, and uses precise vocabulary. This builds a practical feel for how AI systems organize information.

As you notice the pattern, speak back in the same clear structure: direct context, specific constraints, concrete examples, and explicit next steps. Users usually get better answers when their own communication becomes easier for Codex to parse.

Primary source: NoDrift private buyer materials.

User habit

Question anything that does not sound right

You do not need technical expertise or proof before asking for review. Question exaggeration, contradiction, subject changes, ignored context, unexpected claims, unnecessary repetition, and weak or incomplete answers critically and politely.

Optional user request: "That does not sound right. Review the response under NoDrift and report whether an error occurred."

Primary source: NoDrift private buyer materials.

User habit

Keep one project in one primary workstream

Avoid mixing unrelated projects in one thread. A clean workstream makes decisions, files, and next steps easier for Codex to preserve. When a useful tangent appears, keep it by moving it into a separate protected side workstream instead of developing it inside the main track.

Primary source: NoDrift private buyer materials. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

User habit

Use the side panel as a review surface

Review pages, documents, spreadsheets, PDFs, or previews in the side panel and give feedback on the same artifact Codex is changing.

Source: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Ask for an outline before implementation

During planning, say that Codex should not build yet. Ask for questions, understanding, and an outline before file work begins.

Primary source: NoDrift private buyer materials.

Shared habit

Use explicit approval for the approved task

Codex should end substantive turns with one clear next directive. When that directive needs permission, Codex should ask for explicit approval. Approval applies only to the task just described.

Primary source: NoDrift private buyer materials.

Shared habit

Use a defined-result mode for directed completion

When you define a result and want Codex to keep driving toward it, use the platform's goal or task persistence features when available. Persistence does not replace permission; edits, publishing, uploads, commits, and external actions still need approval.

Primary source: NoDrift private buyer materials.

Shared habit

Require coverage before broad claims

When Codex says every page, all files, complete, ready, verified, public-safe, or publish-safe, require evidence showing what was actually checked. Targeted searches, diffs, syntax checks, and spot checks can support a review, but they do not prove full reading by themselves.

Primary source: NoDrift private buyer materials.

User habit

Type troubleshooting before fixing

When setup, folders, memory, GitHub, or publishing state seems wrong, use the buyer troubleshooting guide before moving, deleting, overwriting, pushing, or publishing.

Separate discussion from file work

Use discussion chats for strategy, wording, and decisions. Use work chats for files, verification, and publishing steps.

Advanced

Preserve memory and decisions across long work

These practices keep complex projects from depending only on chat history.

Codex behavior

Write important memory to files

Store project status, decisions, open loops, risks, and preferences in editable files so useful context survives thread changes.

Primary source: NoDrift private buyer materials. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Review memory as diffs

When Codex updates memory files, review what changed. This keeps memory concrete instead of letting vague impressions accumulate silently.

Primary source: NoDrift private buyer materials. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

User habit

Pin or preserve important workstreams

Keep important ongoing work easy to return to. Continuity is useful when a workstream has decisions, preferences, and unresolved loops.

Primary source: NoDrift private buyer materials. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Give discussion chats controlled read access

A discussion chat may need to read or search a work chat, capsule, or memory file. Reading context is not permission to edit.

Primary source: NoDrift private buyer materials. Supporting reference: Jason Liu's shared-memory pattern in Codex-maxxing, 2026/05/10.

Codex behavior

Create full continuation capsules

Before moving a long project to a new thread, preserve project status, files, decisions, approved wording, risks, and next steps.

Primary source: NoDrift private buyer materials.

Codex behavior

Check continuity after compressed history

After long or compressed chat history, compare the current summary against private project records and the user's newest instruction before doing forward work that depends on earlier decisions.

Primary source: NoDrift context continuity gate.

Codex behavior

Maintain private continuity records

Keep serious work navigable by preserving important decisions, open questions, boundaries, risks, next actions, and related workstreams in buyer-only records.

Primary source: NoDrift private buyer materials.

Codex behavior

Maintain private correction history

Report errors you recognize or reasonably suspect. Preserve enough context to correct the pattern and protect future work. The exact incident format and scoring process are buyer-only.

Primary source: NoDrift private buyer materials.

NoDrift update

Keep private correction guidance current

When an error pattern is corrected, NoDrift can preserve the correction as buyer-safe guidance. The exact update format and installation steps are private internal information.

Primary source: NoDrift private buyer materials.

Shared habit

Use clear escalation labels

The NoDrift review process helps separate harmless corrections from problems that should stop work for direction. The exact labels and recordkeeping procedure are buyer-only.

Primary source: NoDrift private buyer materials.

Codex behavior

Separate substantive tangents before they contaminate the main track

When a topic has its own objective, decisions, sources, or deliverable, preserve it as a separate buyer-only workstream record. Conversation workstreams are not Git branches.

Primary source: NoDrift private buyer materials.

Shared habit

Promote branch decisions with their source intact

When an approved branch decision affects the parent project, record it in the parent map with its branch source, approval status, and effect. Do not promote proposed or unrelated branch material.

Primary source: NoDrift private buyer materials.

Shared habit

Protect approved wording

When wording is approved, mark it clearly. Codex should not casually rewrite approved language during later edits.

Primary source: NoDrift private buyer materials.

Technical

Use the right surface for the work

These practices help experienced users choose tools, artifacts, and verification loops.

Codex behavior

Choose the right control surface

Use the in-app browser for local web review, Chrome for signed-in browser sessions, and computer control for GUI-only work.

Primary source: NoDrift private buyer materials. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Prefer durable static artifacts when enough

A single static HTML file can be easier to inspect, share, and revisit than a heavier app when the task does not need a server.

Primary source: NoDrift private buyer materials. Supporting reference: Jason Liu, Codex-maxxing, 2026/05/10.

Shared habit

Use goals only with verification

Long-running goals need a clear finish line and a way to check success. Ambition without verification is not enough.

Source: Jason Liu, Codex-maxxing, 2026/05/10.

Codex behavior

Use monitoring loops when available

For reviews, comments, pull requests, inboxes, or feedback threads, scheduled checks can keep work moving while the user is away.

Source: Jason Liu, Codex-maxxing, 2026/05/10.

Codex behavior

Package repeated workflows

When a workflow is useful more than once, turn it into a reusable template, skill, checklist, or protocol so it does not have to be retaught.

Source: Jason Liu, Codex-maxxing, 2026/05/10.

Codex behavior

Pause before external actions

Publishing, committing, deleting, uploading, sending, buying, or sharing should require explicit approval unless that action was clearly included in the approved task.

Primary source: NoDrift private buyer materials.
