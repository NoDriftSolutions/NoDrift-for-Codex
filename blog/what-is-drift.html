Drift and correction

What Is Drift?

Drift is when the user asks a question and the AI agent's response changes the topic slightly or a lot, leaving the user to manage a different direction than the one they asked for.

Reading depthFoundation

The problem in plain language

You ask one thing. The AI agent's response begins close enough to sound useful, then slides into a nearby topic, a bigger plan, a different assumption, or a polished response to a question you did not ask.

That is drift. Sometimes it is small. Sometimes it changes the whole shape of the work. From the user's side, it can feel like the response is steering away from the actual question. The LLM has no human intent, but the user can still experience the turn as if it is steering away.

A real example

You ask, "Which file controls the approval wording?" The answer starts by naming approval wording, then gives a broad plan for rewriting the workflow, updating the website, and improving the product structure. Some of that may be interesting. It is still not the answer to the question.

The cost is not just annoyance. Drift burns attention, time, and tokens. It can also cause the user to approve the wrong work because the conversation quietly moved to a different object.

How NoDrift treats drift

NoDrift does not control the LLM. It works in the user-side project workspace, where it governs how the AI agent receives, interprets, checks, corrects, and carries forward received model information.

When drift appears, the useful move is to name it early. A user can say something like: "Heartbeat drift. Answer the exact question I asked. Do not widen the task." That kind of check pulls the AI agent back to the user's actual object.

Practical user hint

When an answer starts changing the subject, do not argue with the whole new direction. Stop the turn, name the drift, and restate the exact question. The earlier drift is caught, the less repair the project needs.

The communication standard

The strongest way to handle received model information is to be critical and literal. Do not reward ambiguity, innuendo, exaggeration, underreporting, overreporting, or lateral movement away from the question.

Good communication should move in exact increments: the user gives an idea, the AI agent hears it clearly, adds only what is needed for the user to continue, and the exchange moves back and forth without changing the object. NoDrift works best when the user practices that same quality of communication in the information given to the agent.

When the agent explains its reasoning or work path, read the structure of that explanation carefully. Use the same discipline in your own instructions: exact object, exact boundary, exact next step, and no hidden permission.

Safe boundary

NoDrift does not guarantee that drift will never happen. It gives the user a way to notice, name, correct, and record the boundary so the next turn has a better chance of staying tied to the real question.
