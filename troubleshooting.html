When something seems wrong, type troubleshooting in Codex

NoDrift troubleshooting happens inside the user's workspace. The public page shows the support model; the exact diagnostic prompts and checklists are buyer-only.

The simple trigger

If something goes wrong while you are already working in your NoDrift workspace, send this to Codex:

troubleshooting

The buyer guide gives the exact diagnostic wording. Codex should pause normal work, narrow the problem, inspect only what is needed, explain what it found, and ask for approval before changing, moving, deleting, restoring, publishing, pushing, uploading, or editing anything.

Stop points

What Codex must not do without approval

Troubleshooting is allowed to inspect within scope. Consequential changes still need approval.

File safety

Do not delete or overwrite first

Duplicate-looking files, old folders, C drive copies, G drive copies, and GitHub versions must be inventoried before anything is deleted, replaced, or moved.

External actions

No publish, push, upload, or send by implication

Access is not approval. Local preview is not the public website. GitHub repository state is not the same thing as GitHub Pages publication.

Private records

Keep memory private

Private continuity records, correction records, source records, and working notes are not public website material unless the user explicitly authorizes external use.
